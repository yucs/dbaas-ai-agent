# DBAAS 智能助手记忆与压缩策略

## 1. 文档目的

本文档用于单独维护 DBAAS 智能助手的“记忆与压缩”策略。

本项目基于 DeepAgent 实现，因此这里讨论的“记忆与压缩”，
并不是一个独立的长期记忆系统，而是服务于 DeepAgent 在长会话中的上下文恢复。

换句话说：

- 这些记忆内容主要用于帮助 Agent 恢复当前 Session 的执行上下文
- 它们会与 Session、`thread_id`、checkpoint 和审批恢复配合使用
- 它们不负责保存 DBAAS 资源的实时状态

## 1.1 能力边界

在记忆与压缩这一层，需要区分：

DeepAgent 原生支持：

- 短期对话状态随执行线程延续
- long-term memory 的基础机制
- context engineering 的运行时能力
- 与 checkpoint 配合的上下文恢复基础

本项目需要自己开发：

- “只保留操作记录”的记忆策略
- Session 摘要的结构定义
- 哪些内容应保留、哪些不应保留
- 何时触发压缩
- 压缩结果如何落到本地 Session 文件中
- 页面加载 Session 时如何组装上下文

之所以拆成独立文档，是因为这部分很可能随着产品阶段、模型效果、
上下文成本和操作风险不断调整。将其从总设计文档中拆出，有利于：

- 灵活调整策略而不频繁改动总设计
- 明确哪些内容属于会话记忆，哪些不属于
- 统一压缩、恢复、审计的讨论口径
- 为后续实现提供单一参考来源

总设计仍以 [DESIGN.md](./DESIGN.md) 为主，本文件负责细化记忆与压缩细则。

## 2. 当前结论

第一阶段只做“操作记录记忆”，不做泛化长期记忆系统。

这意味着：

- 记录会话中的关键操作事实
- 记录审批与任务状态变化
- 对历史对话做摘要压缩
- 不将 memory 作为资源实时状态的事实来源

以下信息必须实时查询后端，而不是从 memory 中读取：

- 服务当前状态
- 主机当前状态
- 集群当前状态
- 站点当前状态
- 异步任务的最新状态

## 3. 记忆边界

### 3.1 应纳入记忆的内容

- 用户的 DBAAS 操作意图
- 已查询过的资源对象及关键观察结论
- 已提议的工具调用
- 审批请求及审批结果
- 已执行的写操作
- 异步任务创建记录
- 异步任务的重要状态变化
- 当前会话未完成事项

### 3.2 不应纳入记忆的内容

- 实时资源详情原文长期缓存
- 大段无结构的工具返回原文
- 非必要的闲聊内容
- 第一阶段的长期用户偏好
- 与 DBAAS 执行无关的知识型记忆

## 4. 记忆模型

### 4.1 设计原则

记忆采用“事件 + 摘要”的组合模型：

- 事件用于保留关键操作事实
- 摘要用于压缩长对话上下文

推荐组合：

- `session_events` 作为操作事实主记录
- `session_summaries` 作为压缩后的上下文快照

### 4.2 Session 与 Thread

建议保持以下映射关系：

- 一个 `session_id` 对应一个用户会话
- 一个 `thread_id` 对应一个 Deep Agent 执行线程
- 第一阶段采用一个 `session` 对应一个 `thread`

这样可以简化以下能力：

- 会话恢复
- 审批中断与恢复
- 历史加载
- 会话压缩

## 5. 事件模型

### 5.1 候选事件类型

建议保留以下核心事件类型：

- `user_request`
- `agent_response`
- `resource_observation`
- `tool_call_proposed`
- `approval_required`
- `approval_decision`
- `tool_call_executed`
- `task_created`
- `task_status_changed`
- `session_summary_updated`

### 5.2 事件保留原则

- 只保留对执行决策有价值的事件
- 写操作相关事件必须完整保留
- 审批相关事件必须完整保留
- 异步任务关键状态变更必须保留
- 普通闲聊消息不需要长期保留为结构化事件

### 5.3 推荐事件载荷

不同事件的载荷格式后续可以继续收敛，但建议至少包含：

- `event_type`
- `session_id`
- `thread_id`
- `message_id` 或 `run_id`
- `created_at`
- `payload`

其中 `payload` 建议按事件类型结构化，而不是简单保存整段文本。

## 6. 压缩策略

### 6.1 压缩目标

压缩的目标是减少上下文噪音，而不是丢弃操作事实。

压缩主要针对：

- 重复性自然语言对话
- 已经完成且无须原文保留的解释性内容
- 对最终执行无影响的闲聊上下文

### 6.2 不可压缩丢失的内容

以下内容即使进入摘要，也必须能够追溯：

- 写操作提议
- 审批结果
- 已执行操作
- 异步任务 ID
- 关键失败原因
- 当前未完成事项

### 6.3 推荐压缩方式

建议采用“最近原文 + 历史摘要”的模式：

- 保留最近一段原始消息
- 将更早消息压缩为结构化摘要
- 操作事件单独保存，不依赖对话原文

这意味着压缩后仍然可以：

- 恢复执行上下文
- 展示历史操作轨迹
- 审计关键操作

## 7. 摘要结构

建议 Session 摘要统一采用结构化字段，而不是自由文本大段总结。

推荐字段：

- `current_goal`
- `observed_resources`
- `completed_actions`
- `approved_actions`
- `rejected_actions`
- `pending_items`
- `constraints`

推荐示意：

```json
{
  "current_goal": "排查 mysql-xf2 并评估是否需要升级镜像",
  "observed_resources": [
    "service:mysql-xf2 health=DEGRADED"
  ],
  "completed_actions": [
    "queried service mysql-xf2"
  ],
  "approved_actions": [],
  "rejected_actions": [],
  "pending_items": [
    "waiting for human approval on image upgrade"
  ],
  "constraints": [
    "write operations require approval",
    "live state must come from backend APIs"
  ]
}
```

## 8. 恢复时的上下文装配

恢复 Session 时，不建议把全部历史消息都重新喂给模型。

建议只装配以下内容：

1. 系统提示词
2. 当前用户与权限上下文
3. 最新 `session_summary`
4. 最近若干条原始消息
5. 当前未完成审批
6. 当前未完成异步任务

这样做的好处是：

- 恢复更快
- 上下文更干净
- 不容易把旧噪音重新引回模型

## 9. 可调参数建议

本文档建议把以下内容视为后续可调策略项：

- 最近保留的原始消息数量
- 触发压缩的消息阈值
- 摘要生成的时机
- 哪些只读查询需要沉淀为事件
- 异步任务状态记录的粒度
- 审批结果在摘要中的保留方式
- 前端历史展示读原始消息还是读投影摘要

这些参数第一阶段不必一次定死，可以在实现前先给默认值，
后续通过配置或策略文档迭代。

## 10. 与实现的关系

本文档主要定义策略，不直接规定代码细节。

但它会直接影响以下模块设计：

- `sessions/`
- `agent/summarizer.py`
- `agent/context.py`
- `persistence/`
- `api/routes_sessions.py`

后续进入实现阶段时，应保证代码与本文档保持一致；
如果策略发生变化，应优先更新本文档，再调整实现。

## 11. 当前不做的内容

第一阶段暂不包含：

- 长期用户画像记忆
- 跨 Session 的知识型记忆
- 自动学习用户偏好
- 将完整工具返回全文长期缓存到模型记忆中
- 把实时 DBAAS 状态作为 memory 使用

## 12. 后续可继续讨论的问题

后续可以继续收敛这些问题：

1. 是否需要单独区分“审计事件”和“上下文事件”？
2. 异步任务状态是否需要后台轮询后自动写入事件？
3. 摘要是每轮更新，还是达到阈值后更新？
4. 摘要生成是完全自动，还是对关键写操作后强制刷新？
5. Session 恢复时，最近原文保留几轮最合适？
