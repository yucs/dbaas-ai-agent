# DBAAS 智能助手前端页面需求

## 1. 文档目的

本文档用于单独收敛前端页面的产品需求、交互规则和当前阶段边界。

它关注的是：

- 页面应该长什么样
- 用户如何完成登录、查看 Session、继续问答
- 当前阶段哪些能力需要展示
- 当前阶段哪些能力先不要出现在页面上

它不替代后端接口设计和总设计文档，而是作为前端实现的直接基线。

## 2. 前端设计原则

### 2.1 页面先服务当前 MVP

当前阶段前端的核心目标不是做复杂工作台，而是稳定完成下面几件事：

- 简单登录
- 查看当前用户自己的 Session 列表
- 创建 Session
- 打开某个 Session
- 在当前 Session 中继续问答
- 删除 Session

### 2.2 页面不要过早暴露后续能力

当前页面不应提前暴露后续阶段才会稳定的能力，例如：

- 归档 / 恢复
- 审批中心
- SSE 流式事件面板
- DBAAS 工具调用细节
- 任务面板

这些能力后续可以补，但当前不应成为页面主路径的一部分。

### 2.3 页面只认产品层 Session

前端页面只围绕 `session_id` 工作，不直接理解 DeepAgent 内部状态。

也就是说：

- 页面切换的是 `session_id`
- 后端延续的是 `thread_id`

前端不需要直接展示 `thread_id`，也不应该直接依赖运行时内部结构。

### 2.4 页面默认按当前登录用户隔离

页面只显示当前登录用户自己的 Session。

即：

- 当前用户登录后，只看到 `user_id` 对应目录下的 Session
- 不能在页面上看到其他用户的 Session
- 管理员当前阶段也不例外

## 3. 页面范围

当前阶段只需要两个页面形态：

1. 登录页
2. 对话页

## 4. 登录页需求

### 4.1 页面目标

登录页用于建立当前前端用户身份上下文。

当前阶段不接统一认证系统，只做本地轻量登录。

### 4.2 页面元素

登录页只需要：

- 用户名输入框
- 用户类型选择框
  - `user`
  - `admin`
- 登录按钮

### 4.3 行为要求

- 输入用户名并选择用户类型后进入对话页
- 登录信息写入本地存储
- 页面刷新后可以恢复当前登录状态
- 只有在“未登录”时才显示登录弹层
- 登录弹层默认应隐藏，避免页面重绘时闪出

### 4.4 当前阶段身份规则

当前前端登录态至少包括：

- `user_id`
- `role`
- `user`

规则：

- 当 `role = user` 时
  - `user = user_id`
- 当 `role = admin` 时
  - `user` 可以为空

## 5. 对话页需求

### 5.1 页面结构

对话页当前只需要以下结构：

1. 左侧边栏
   - 当前身份信息
   - 新建 Session 按钮
   - 当前用户自己的 Session 列表

2. 主对话区域
   - 当前 Session 标题
   - 当前 Session 状态
   - 删除按钮
   - 消息列表
   - 输入框
   - 发送按钮

### 5.2 左侧 Session 列表

列表中每个 Session 卡片建议展示：

- 标题
- 最近一条预览
- 最近消息时间
- `打开` 按钮
- `删除` 按钮

当前阶段不展示：

- `归档`
- `恢复`

### 5.3 当前 Session 头部

头部当前保留：

- 当前 Session 标题
- 状态展示
- 删除按钮

当前阶段不保留：

- 归档按钮
- 恢复按钮

## 6. 交互流程

### 6.1 登录后进入对话页

流程：

1. 页面恢复本地登录态
2. 请求当前用户的 Session 列表
3. 如果有 Session，则默认加载第一个
4. 如果没有 Session，则自动创建一个新 Session

### 6.2 Session 切换

用户点击某个 Session 后：

1. 加载该 Session 详情
2. 将历史消息展示到当前窗口
3. 后续发送问题时继续使用该 Session

### 6.3 新建 Session

用户点击“新建会话”后：

1. 调用创建接口
2. 刷新当前用户 Session 列表
3. 自动切换到新建的 Session

### 6.4 删除 Session

用户点击删除后：

1. 弹确认框
2. 用户确认后再调用删除接口
3. 删除成功后刷新左侧列表
4. 如果删除的是当前会话，则自动切换到剩余第一个 Session
5. 如果已经没有 Session，则主区域显示空状态

### 6.5 发送消息

当前阶段发送消息的交互要求如下：

1. 用户点击发送
2. 立即在页面中显示用户消息
3. 立即显示一条“助手思考中”占位消息
4. 按钮变为 `发送中...`
5. 等待后端返回完整回答
6. 用真实 assistant 消息替换掉占位消息

### 6.6 发送时的体验要求

当前阶段虽然还没有 SSE，但页面体验应尽量避免“完全卡住”的感觉。

因此需要：

- 本地先插入用户消息
- 本地先插入 assistant 占位消息
- 不在成功后立刻做不必要的全量页面刷新
- 不在发送时弹出登录页
- 只有明确错误时才显示错误提示

## 7. 页面提示规则

### 7.1 哪些提示可以显示

当前阶段页面可以显示轻量提示条，用于：

- 一般错误提示
- DBAAS 问题命中但后台尚未启用
- 初始化失败

### 7.2 哪些提示不应出现

当前阶段不应在发送消息时出现：

- 登录弹层重新弹出
- 切换用户弹层误显示
- 非删除场景下的浏览器确认框

### 7.3 删除确认的唯一例外

当前页面允许出现浏览器确认框的场景只有一个：

- 删除 Session

发送消息、切换 Session、加载历史消息都不应触发确认弹窗。

## 8. 当前阶段不做的前端能力

以下能力明确后置：

- 归档 / 恢复 UI
- 审批卡片
- SSE 流式输出
- Token 级逐字渲染
- DBAAS 工具执行进度面板
- 多标签页同步
- 搜索 Session
- 收藏 Session
- Session 分组

## 9. 与后端接口的关系

当前前端主要依赖以下接口：

- `GET /api/v1/sessions`
- `POST /api/v1/sessions`
- `GET /api/v1/sessions/{session_id}`
- `POST /api/v1/sessions/{session_id}/messages`
- `DELETE /api/v1/sessions/{session_id}`

当前阶段前端不依赖：

- `archive`
- `restore`
- SSE 事件流接口
- 审批接口

## 10. 当前阶段验收标准

达到下面这些标准，即可认为当前前端页面符合需求：

1. 未登录时显示登录页
2. 登录后进入对话页
3. 左侧只显示当前用户自己的 Session
4. 可以新建、打开、删除 Session
5. 删除时有确认框
6. 发送消息时会立即看到用户消息和“助手思考中”
7. 发送消息时不会弹出登录页
8. 当前页面不显示归档 / 恢复
9. 刷新页面后仍能恢复当前登录态
10. 选择历史 Session 后可以在当前窗口继续问答
