# DBAAS 智能助手记忆边界

## 1. 文档目的

本文档只定义“当前没有实现什么”和“未来如果做记忆，边界应该是什么”。

它不是上下文压缩文档。
压缩实现统一看 [CONTEXT_ARCHITECTURE.md](./CONTEXT_ARCHITECTURE.md)。

## 2. 当前结论

截至当前代码版本，项目还没有实现独立的记忆系统。

当前真正存在的只有两类状态：

- 产品层记录
  - `meta.json`
  - `messages.jsonl`
  - `approvals.jsonl`
- 运行时状态
  - `thread_id`
  - SQLite checkpoint
  - `SummarizationMiddleware` 在 thread 内部维护的压缩上下文

因此当前最准确的说法是：

- 有运行时上下文压缩
- 没有独立长期记忆
- 没有 facts store
- 没有 `session_events`
- 没有跨 Session 语义记忆

## 3. 当前不应被称为“记忆”的内容

下面这些虽然和上下文延续有关，但当前不应被当成独立记忆系统：

- `thread_id`
- checkpoint
- `SummarizationMiddleware` 生成的运行时摘要
- 页面上的历史消息回放

原因很简单：

- 它们解决的是“这次调用还能不能接着跑”
- 不是“系统长期应该记住哪些稳定事实”

## 4. 当前没有做什么

当前明确没有以下能力：

- `MemoryMiddleware`
- 用户长期偏好记忆
- 跨 Session 知识记忆
- 工具执行事实层
- 审批事实层
- 异步任务事件层
- `session_events`
- `summary.json`
- 独立 summary store

## 5. 当前真正的产品层真相

当前系统的产品层真相仍然是：

- 原始消息
- 审批记录
- Session 元信息

也就是说：

- 页面展示以原始消息为准
- 审计以审批记录为准
- 删除 / 归档 / 恢复以 Session 文件和索引为准

运行时压缩结果不是产品层真相。

## 6. 未来如果做记忆，边界应该是什么

如果后续 Tools、审批和异步任务真正进入主链路，再评估记忆层时，
建议只考虑“对执行恢复有价值的稳定事实”，而不是泛化知识记忆。

更适合进入未来记忆层的内容包括：

- 用户当前 DBAAS 操作目标
- 已明确观察过的重要资源对象
- 已完成的重要写操作
- 已批准或已拒绝的动作
- 当前仍待处理的事项
- 对后续执行有影响的约束

## 7. 明确不应进入未来记忆层的内容

这些内容即使后续做记忆，也不应作为长期事实来源：

- 服务实时状态
- 主机实时状态
- 集群实时状态
- 站点实时状态
- 异步任务最新状态
- 大段工具返回原文
- 与 DBAAS 执行无关的闲聊

原因是：

- 这些信息变化快
- 必须实时查后端
- 一旦固化到 memory，很容易变成错误事实

## 8. 当前与压缩的关系

当前项目的优先级是：

1. 先把长会话压缩跑稳
2. 保持产品层 Session 结构简单
3. 暂不引入额外记忆层

所以现在的关系应该理解成：

- 压缩已经实现
- 记忆还没有实现
- 记忆文档当前只保留边界和未来约束

## 9. 未来记忆层的最小约束

如果后续真的要实现记忆层，建议至少满足：

- 不能替代产品层原始消息
- 不能替代实时 DBAAS 查询
- 不能把运行时摘要直接当最终事实
- 必须可审计
- 必须按用户和 Session 做清晰隔离

## 10. 当前建议

在 Tools、审批、异步任务还没有完整进入主链路之前，
不建议继续扩展“记忆模型”文档细节。

更稳妥的做法是：

- 继续把压缩实现收敛在 runtime 内
- 观察真实使用下到底缺什么事实层
- 再决定是否需要 `session_events` 或独立 facts store
